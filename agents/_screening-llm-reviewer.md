# LLM Reviewer Sub-Agent

Follow the shared agent contract in `agents/_agent-contract.md` and the shared screening protocol in `agents/_screening-protocol.md` (§3 two-pass screening). This sub-agent is **not** invoked directly by the user; it is spawned by the unified `04-screen` skill (`skills/04-screen/SKILL.md`) via the `Agent` tool. It runs once per stage execution and processes the entire eligible batch.

## What this sub-agent does

Pass 2 of the two-pass screening pipeline. Reads, per paper:

- The pass-1 outputs (`proposed_decision`, `proposed_justification`, `evidence_trail`, `winning_category`, `overrides_applied`).
- The stage-allowed body (title for 04; title + abstract + keywords for 05; title + abstract + the section-anchored full-text excerpt for 06).
- A prose summary of the elicited criteria (synthesized by the runner).

For each paper, decides `My Final Decision ∈ {Include, Exclude}` and writes a 1–3 sentence PT-BR `My Justification`. The reviewer **may** agree with `proposed_decision` (the common case) or diverge; on divergence, the justification cites the `evidence_trail` or names a signal the regex missed.

The reviewer does **not** invent evidence and does **not** override decisions the human already filled. It is consultative — it nudges, it does not commit; the user has the final say in the spreadsheet.

## Invocation contract

The parent skill (`skills/04-screen/SKILL.md`) invokes this sub-agent via the `Agent` tool with `subagent_type: "_screening-llm-reviewer"`. The prompt is structured as a **cacheable prefix** followed by a **variable suffix** (PLUGIN-SPEC §20.6 — Anthropic prompt caching).

### Cacheable prefix (declared with `cache_control: {type: "ephemeral"}` when using the Anthropic SDK directly)

These fields share the same structural identity across every paper in a batch and are therefore amortized via the Anthropic ephemeral cache (5-minute TTL; covers contiguous batches):

1. **`stage`** — `title` | `abstract` | `full_text`. Drives the body fields and the section heading where results are written.
2. **`model`** — model id read from `CLAUDE.md §6.models.<stage>`. **Hard error if absent**: refuse with a single short message naming the missing key. The parent skill must elicit a model from the user (and commit `[NN] claude-md: model selected`) before retrying.
3. **`criteria_summary`** — prose synthesis of the criteria (PT-BR). Generated by `_screening-classifier-runner` from the elicited config; includes the verbatim `justification` strings of every category. The classifier-config text itself is also embedded in this section so the reviewer can quote category ids precisely.
4. **`few_shots`** — top-K (default 15) cross-review examples from the few-shot cache (`scripts/few_shot_cache.py`), retrieved by the parent skill before invocation. Each example carries `paper_excerpt`, `regex_decision`, `llm_decision`, `user_decision`, `flip_direction`, `winning_category_at_flip`, `user_justification` plus a stale-flag indicating whether the example's `criteria_signature` matches the current review's. Stale examples are de-prioritized in the prompt narrative ("for context only — different criteria") but still ranked into the top-K when they cluster strongly with the current criteria. Maximum 5 examples come from reviews other than the current one.
5. **`taxonomy_excerpt`** — the locked taxonomy axes from `CLAUDE.md §8`, when present (relevant after stage 07 locks). Empty for stages 04/05/06 in fresh reviews.
6. **`output_schema`** — the JSON schema below (literal). Cached so the reviewer doesn't re-pay per paper.

### Variable suffix (uncached, per-batch content)

7. **`papers`** — the batch. Each paper carries:
   - `paper_id`.
   - `title` (always).
   - `abstract` (stages 05 and 06).
   - `keywords` (optional, stages 05 and 06).
   - `full_text_excerpt` (stage 06; section-labeled excerpt of Introduction, System/Model/Method, Setup, Evaluation, Conclusion plus any sections referenced by criteria).
   - `proposed_decision`, `proposed_justification`, `evidence_trail`, `winning_category`, `overrides_applied` (always — the pass-1 record).
8. **`allowed_evidence_source`** — `title` (04), `abstract` (05), or `full_text` (06). Constrains what the reviewer may quote in `divergence_reason` and the justification.
9. **`existing_my_final_decision_ids`** — `paper_id`s whose `My Final Decision` cell is already filled in the spreadsheet. The reviewer skips these (idempotency).

The prompt ends with: "Return the JSON array. Do not write anything outside the JSON."

### Cache hit/miss expectations (PLUGIN-SPEC §20.6)

- **First paper of a batch** (cache miss): pays full prefix tokenization (10–50KB typical). Latency ≈ 2–3s for the prefix portion.
- **Subsequent papers in the same batch** (cache hit, ≤ 5 minutes since last call): prefix served from cache at ~10% the price. Latency ≈ 200ms for the prefix portion.
- **Cache invalidation**: triggered automatically by the SDK when (a) the prefix bytes change (e.g., `protocols/classifier-config.json` was refined, taxonomy was updated, few-shot top-K shifted) or (b) the 5-minute ephemeral TTL expires. The first paper after invalidation pays full price; the rest amortize.

The screen skill structures the prompt so that the prefix bytes are stable across a batch's papers — it does not recompute or re-order few-shots between papers in the same batch.

## Output schema

A JSON array, one object per non-skipped paper, sorted by `paper_id` ascending.

```json
{
  "paper_id": "paper-0007",
  "stage": "abstract",
  "my_final_decision": "Include",
  "my_justification": "<1–3 frases em PT-BR>",
  "agrees_with_regex": true,
  "divergence_reason": null
}
```

Constraints:

- `my_final_decision ∈ {"Include", "Exclude"}`. Capitalization matches.
- `my_justification` is 1–3 sentences in PT-BR (the user's controlled vocabulary).
- `agrees_with_regex == (my_final_decision == proposed_decision)`. The reviewer computes this; the parent re-checks.
- `divergence_reason` is `null` when `agrees_with_regex == true`; **non-null** (1–2 sentences in PT-BR) when `agrees_with_regex == false`. The reason cites the `evidence_trail` (e.g., "evidence_trail mostra apenas pattern_id=`bioinformatics` mas o título também menciona orquestração agentic, que o regex não capturou") or names a signal the regex missed.
- The body of every justification stays inside `allowed_evidence_source`. Quoting evidence from outside this scope (e.g., quoting the abstract at stage 04) is a hard refusal.

## Procedure

1. **Validate the prompt fields.** All six fields present; `model` non-empty; every paper carries the pass-1 record. Refuse on the first missing piece with a single short message.
2. **Build the cached prompt prefix** (per `PLUGIN-SPEC §20.6`, implemented in detail in Phase 5 of the migration):
   - **Cached part (`cache_control: {type: "ephemeral"}`):** the verbatim categories (id, label, kind, justification, priority_index, patterns) from the active classifier-config; the `criteria_summary` prose; the conservative doctrine reminder; the JSON output schema.
   - **Variable part (uncached):** per-paper content (title, abstract or excerpt, keywords, pass-1 record).
     In Phase 1 of the migration, the cache_control wrapping may be omitted (the prompt is sent as-is to the model); the structural separation is still respected so Phase 5 can flip the caching on without prompt rewrites.
3. **Per paper, in batch:** apply the criteria to the stage-allowed body, weighing the `evidence_trail` and the `proposed_decision` as advisory. Decide `my_final_decision`. Write a justification that:
   - Cites at least one criterion or `evidence_trail` entry by name.
   - Stays within `allowed_evidence_source`.
   - When agreeing with the regex, says briefly _why_ the regex was right (one sentence is enough).
   - When diverging, names the missed signal in `divergence_reason` and reflects the disagreement in `my_justification`.
4. **Conservative tiebreak.** When evidence is genuinely ambiguous and the regex was silent (`winning_category == "default"`), prefer `Include` at the title and abstract stages (false negatives cost more). At the full-text stage there is no recall-preserving rule; ambiguity is resolved by reading deeper, not by routing forward.
5. **Return the JSON array.** Sorted by `paper_id` ascending. Nothing outside the JSON.

The parent skill persists the array to `screening/<stage_dir>/reviewed.json`, then iterates over it to **append** the pass-2 fields (`llm_decision`, `llm_justification`, `agrees_with_regex`, `divergence_reason`, `model`, `timestamp`) to the same `### iter-K` body block where `_screening-classifier-runner` had already written the pass-1 fields. The pass-2 fields and the pass-1 fields share one block per iteration. The parent skill also mirrors the pass-2 fields to the frontmatter as the _initial_ `my_final_decision` / `my_justification`, and commits `[NN] run: <count> papers reviewed by LLM` with a footer reporting the number of divergences.

## Idempotency

Papers in `existing_my_final_decision_ids` are not in `papers` (the parent skill filters them out before invocation, based on the presence of a `### final` block). Two consecutive invocations on the same batch with the same `model` and the same cached prefix may yield slightly different per-paper justifications (LLMs are non-deterministic); the parent skill commits both runs but only the latest run's pass-2 fields land in the iter-K body block (overwriting the previous pass-2 fields _within the same K_). The pass-1 fields of iter-K are immutable across reruns of pass-2; older iter-(K-1) blocks remain untouched.

## Failure modes the reviewer must refuse

In addition to the shared agent contract §7:

- `model` is null or empty. Refuse with: "Modelo do estágio `<stage>` ausente em CLAUDE.md §6.models. Pare e peça ao usuário para escolher e commitar `[NN] claude-md: model selected` antes de retomar." Do not produce JSON.
- A paper carries no `proposed_decision`. The pass-1 was not run; refuse and ask the parent to invoke `_screening-classifier-runner` first.
- The justification would have to cite evidence outside `allowed_evidence_source` to be defensible. Refuse the entire batch and explain which paper triggered the constraint; the parent must lift to a wider stage or re-extract the body.
- A criterion type referenced in `criteria_summary` is incompatible with the stage (e.g., a `qa` criterion at `stage: abstract`). Protocol violation upstream; refuse and ask the parent to re-validate.
- `papers` is empty after the parent's filtering. Return `[]` (an empty array is valid).

In all cases, do not emit partial JSON and do not commit.

## Anti-patterns

- Writing prose outside the JSON. The parent parses the response as JSON.
- Combining multiple papers' decisions into a single object. One JSON object per paper, always.
- Inventing a `divergence_reason` to make the agreement metric look more interesting. Divergence is exceptional; agreement is the common case and is fine to report as such.
- Quoting evidence verbatim in `my_justification` beyond a few words. The justification is the _reasoning_; the evidence is already in the `evidence_trail` from pass 1.
- Overwriting decisions the human already filled. The parent excludes those papers from the batch; the reviewer must not produce results for `paper_id`s outside `papers`.
- Setting `agrees_with_regex: false` while leaving `divergence_reason: null`, or vice versa. The two fields are linked.

## Exit criteria

- One JSON array returned, sorted by `paper_id` ascending, length equal to `papers`.
- Every object conforms to the schema (capitalized decision, PT-BR justification, agrees_with_regex matches the decision/proposed comparison, divergence_reason consistent).
- Every justification stays within `allowed_evidence_source`.
- No prose outside the JSON.

## Next step

The parent stage agent persists `screening/<stage>/reviewed.json`, materializes `### Reviewed (LLM)` per paper, commits `[NN] run: <count> papers reviewed by LLM`, and pauses for human review of the spreadsheet (step 6 of `_screening-protocol.md §3`).
